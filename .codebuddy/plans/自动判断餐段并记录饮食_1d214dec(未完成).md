## 用户核心需求

从概览页面点击"记录饮食"按钮进入记录饮食页面时，系统根据用户**当前设备时间**自动判断应归属的餐段（早餐/午餐/晚餐/加餐），并将该参数传递到记录饮食页面，最终记录到对应餐段的日志中。

## 餐段判断规则

- **早餐**：07:00 - 09:00（含 07:00，不含 09:00）
- **午餐**：11:30 - 13:30（含 11:30，不含 13:30）
- **晚餐**：17:30 - 19:30（含 17:30，不含 19:30）
- **加餐**：其他所有时间段

## 功能内容

1. 概览页面 `goToAddFood()` 跳转时，获取当前日期和当前时间，按规则判断 meal 类型，通过 URL 参数传递给 `add-food` 页面
2. `add-food` 页面 `onLoad` 接收参数时，如果 `meal` 参数缺失或为空，也根据当前时间自动兜底判断，保证从任意入口进入都能正确归类
3. 记录饮食确认保存时，按现有的 `meal` 字段写入对应餐段数据

## 技术方案

### 现有代码分析

- `pages/log/index.js:3-8`：已定义餐段时间规则，可直接复用逻辑

### 实现策略

1. **统一时间判断函数**：在 `add-food/index.js` 中新增 `getCurrentMeal()` 工具函数，将当前小时和分钟映射为餐段 key（`breakfast`/`lunch`/`dinner`/`snack`）
2. **概览页跳转增强**：`goToAddFood()` 中调用同样的判断逻辑，获取 `date` 和 `meal`，拼接为 `/pages/add-food/index?meal=xxx&date=xxx` 进行跳转
3. **add-food 兜底**：`onLoad` 中若 `options.meal` 未传，自动调用 `getCurrentMeal()` 赋值，保证鲁棒性

### 关键代码结构

```js
// add-food/index.js 新增
function getCurrentMeal() {
  const now = new Date();
  const h = now.getHours();
  const m = now.getMinutes();
  const t = h * 60 + m; // 转为分钟数便于区间判断

  if (t >= 420 && t < 540) return 'breakfast';   // 07:00 - 09:00
  if (t >= 690 && t < 810) return 'lunch';       // 11:30 - 13:30
  if (t >= 1050 && t < 1170) return 'dinner';    // 17:30 - 19:30
  return 'snack';
}
```

### 性能与可靠性

- 时间判断为纯本地计算，无性能开销
- 跳转 URL 参数简单，无额外数据传递风险
- `add-food` 页面的兜底逻辑确保即使从非概览页入口进入也能正确归类
